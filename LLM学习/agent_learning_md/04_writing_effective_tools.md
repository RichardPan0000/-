# Writing Effective Tools：Agent 工具设计思想

> 说明：这是“中文学习版”笔记，不是网页全文逐字翻译。内容以原文思想、关键概念、工程实践和学习练习为主。  
> 原文：https://www.anthropic.com/engineering/writing-tools-for-agents

## 一句话理解

这篇文章的核心是：**工具不是 Agent 的附属品，工具设计本身决定 Agent 的上限。**

## 为什么工具重要

Agent 通过工具和外部世界交互。工具如果不好用，模型再强也会出错。

一个差工具的表现：

- 名字含糊。
- 参数不清晰。
- 返回内容太杂。
- 失败信息不明确。
- 多个工具边界重叠。

一个好工具的表现：

- 名字具体。
- 描述明确。
- 参数少而清楚。
- 返回结构稳定。
- 有错误提示和边界说明。

## 工具设计原则

### 1. 工具名要具体

不好：

```text
query()
```

好：

```text
search_project_docs(query: str)
get_customer_order_history(customer_id: str)
```

### 2. 参数要让模型不容易犯错

例如文件操作工具最好要求绝对路径，而不是相对路径。

### 3. 返回值要适合继续推理

工具返回不要只给“成功”，而要给：

```json
{
  "success": true,
  "data": "...",
  "warnings": [],
  "next_suggested_action": "..."
}
```

### 4. 工具文档要像给新人写说明

工具描述应该包括：

- 什么时候用。
- 什么时候不用。
- 输入格式。
- 输出格式。
- 常见错误。
- 示例。

## 对定制开发的启发

你做企业 Agent 时，核心工作之一就是把业务系统能力封装成高质量工具：

- 查数据库工具。
- 查文档工具。
- 写入工单工具。
- 生成报告工具。
- 代码分析工具。
- 运行测试工具。

真正的壁垒不是“会调模型”，而是“会把业务能力变成 Agent 可用的工具接口”。


## 建议练习

1. 用自己的业务场景复述本文核心思想。
2. 写出一个可以落地的小例子，不要只停留在概念。
3. 总结：哪些部分适合固定 workflow，哪些部分适合交给 LLM/Agent 动态决策。
