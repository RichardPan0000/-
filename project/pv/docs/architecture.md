# PV预测业务架构文档

## 1. 系统概述

PV（光伏）预测系统是一个基于机器学习和深度学习的多模型预测平台，用于预测光伏电站的发电功率。系统支持多时区、多区域的并行预测，包含日前预测和日内预测两种场景。

## 2. 核心架构

### 2.1 主调度流程 (`main_schedule`)

主调度器负责整个系统的任务编排和生命周期管理：

```
main_schedule
├── 获取AI区域列表（时区列表）
├── 创建线程池执行器
│   ├── 训练执行器（train_scheduler）
│   ├── 预测执行器（predict_scheduler）
│   └── 更新执行器（update_scheduler）
├── 为每个时区添加定时任务
│   ├── 日前训练任务（每天20:00）
│   ├── 日内训练任务（每天02:00）
│   ├── 日前预测任务（每天19:05）
│   └── 日内预测任务（每天11:35）
└── 动态更新区域列表（每天00:00）
```

**关键特性**：
- 使用 `BackgroundScheduler` 实现定时任务调度
- 支持时区感知，每个区域按其本地时间执行任务
- 动态添加/删除区域，无需重启服务

## 3. 训练流程

### 3.1 日前训练 (`train_area_parallel`)

**执行时间**：每天20:00（本地时区）

**主要步骤**：

1. **获取格点站点**
   - 从数据库获取指定经纬度格点的所有站点
   - 筛选需要训练的站点（基于Redis缓存的训练标记）

2. **加载NWP数据**
   - 时间范围：过去400天到当前日期
   - 数据来源：数据库中的格点天气数据

3. **特征工程**
   - `feature_engineering_xgb5fold_simple_v1`：简单特征（温度、辐照等）
   - `feature_engineering_xgb5fold_simple_v3`：复杂特征
   - `feature_engineering_pca`：PCA降维特征

4. **逐站训练模型**
   - **GridXGBSimpleBayes4fold-dayahead**：4折交叉验证XGBoost模型
   - **GridXGBSimplePCA5Fold-dayahead**：5折交叉验证 + PCA特征
   - 归一化处理：功率值除以 `max_power/20`
   - 模型存储到S3

5. **内存管理**
   - 使用锁机制防止并发问题
   - 训练完成后清理内存

### 3.2 日内训练 (`train_area_intraday_parallel`)

**执行时间**：每天02:00（本地时区）

**流程**：与日前训练类似，但使用日内数据（更高频率更新）

## 4. 预测流程

### 4.1 日前预测 (`predict_area_multi_model_dayahead`)

**执行时间**：每天19:05（本地时区）

**主要步骤**：

```
predict_area_multi_model_dayahead
├── 1. 获取格点列表
│   └── 按批次处理（batch_size=32）
├── 2. 加载NWP数据
│   ├── 历史数据：过去N天（配置的history_days_for_deep_post_opt）
│   └── 未来数据：明天+后天
├── 3. 深度模型预测 (predict_area_deep_dayahead)
│   ├── 准备夏令时信息
│   ├── 获取站点PV历史数据
│   ├── 数据预处理（归一化、特征提取）
│   ├── Timexer模型预测
│   │   ├── 第一天预测
│   │   └── 第二天预测
│   └── 深度后处理优化
├── 4. 最优模型选择（并行）
│   ├── 根据站点数量动态调整线程数
│   │   ├── >3000站点：12线程
│   │   ├── >1000站点：8线程
│   │   └── ≤1000站点：4线程
│   └── 基于历史回测选择最优模型
│       └── select_best_model_v2_dst_wst
└── 5. 输出预测结果到Kafka
```

**深度模型预测详细流程**：

1. **数据收集**
   - 站点信息：从经纬度区域获取所有站点
   - PV历史数据：过去8天的5分钟粒度数据
   - NWP数据：批量加载格点天气数据

2. **数据预处理**
   - 时间对齐：对齐到2304个时间点（8天）
   - 缺失值处理：使用历史同一时刻的平均值填充
   - 特征工程：提取 `r_pm_am`、`r_tail`、`t_cog_norm` 等统计特征

3. **批量预测**
   - 使用 Timexer 深度学习模型
   - 模型：`Timexer-pv-dayahead-mae-mape`
   - 输入维度：(batch_size, 2304, 5)
   - 输出：未来一天的功率预测

4. **后处理优化**
   - 查询历史预测数据
   - 计算历史误差模式
   - 应用后处理算法优化预测结果

5. **最优模型选择**
   - 对比多个模型的历史表现
   - 基于 MAE、RMSE 等指标选择最优模型
   - 将最优预测结果写入 Kafka

### 4.2 日内预测 (`predict_area_intraday`)

**执行时间**：每天11:35（本地时区）

**流程**：
- 调用 `predict_v1_intraday` 进行格点预测
- 使用日内数据（更新频率更高）
- 最优模型选择（支持夏令时处理）

## 5. 数据处理细节

### 5.1 NWP数据处理

**数据来源**：
- 格点化的数值天气预报数据
- 时间粒度：通常为小时级
- 包含字段：温度、辐照度、云量、风速等

**处理流程**：
1. 从数据库按格点批量加载
2. 时间范围：历史N天 + 未来2天
3. 特征工程生成衍生特征
4. 缓存优化（避免重复加载）

### 5.2 PV历史数据处理

**数据来源**：
- 站点实际发电功率
- 时间粒度：5分钟

**处理流程**：
1. 加载站点历史数据（过去8天）
2. 异常值处理（可选的滤波数据）
3. 时间对齐到2304个时间点
4. 缺失值填充策略：
   - 优先使用同一时刻的历史平均值
   - 后备方案：线性插值

### 5.3 特征工程

**XGBoost特征**：
- 时间特征：小时、分钟、星期、月份
- 天气特征：温度、辐照度、云量
- 统计特征：滑动窗口平均、最大、最小值
- PCA降维特征：降低特征维度

**深度学习特征**：
- `r_pm_am`：下午/上午功率比
- `r_tail`：傍晚/正午功率比
- `t_cog_norm`：功率重心时间（归一化）

## 6. 模型架构

### 6.1 XGBoost模型

**模型类型**：
- `GridXGBSimpleBayes4fold-dayahead`：贝叶斯优化 + 4折交叉验证
- `GridXGBSimplePCA5Fold-dayahead`：PCA特征 + 5折交叉验证

**训练策略**：
- 交叉验证防止过拟合
- 归一化处理提高训练稳定性
- 增量训练（定期重训练）

**存储**：S3对象存储

### 6.2 Timexer深度学习模型

**模型特点**：
- 时序预测专用模型
- 支持多变量输入（PV + NWP）
- 输出：未来24小时功率预测

**后处理**：
- 历史误差学习
- 自适应调整预测结果

### 6.3 模型选择策略

**评估指标**：
- MAE（平均绝对误差）
- RMSE（均方根误差）
- MAPE（平均绝对百分比误差）

**选择流程**：
1. 获取所有候选模型的历史预测
2. 计算过去N天的回测误差
3. 综合评分选择最优模型
4. 考虑夏令时和特殊时段

## 7. 系统优化特性

### 7.1 并行处理

- **格点并行**：多个格点同时训练/预测（ThreadPoolExecutor）
- **站点并行**：最优模型选择时并行处理多个站点
- **批量处理**：NWP数据批量加载，减少数据库查询

### 7.2 内存管理

- 分批处理大数据集（batch_size=32）
- 及时释放不再使用的对象（`gc.collect()`）
- 使用生成器避免一次性加载所有数据

### 7.3 容错机制

- Try-catch包裹关键步骤
- 单个站点失败不影响整体流程
- 详细日志记录便于排查问题

### 7.4 夏令时处理

- 自动检测夏令时切换日期
- 处理时间跳变（gap_time）
- 确保预测结果时间戳正确

## 8. 数据存储

### 8.1 输入数据

- **数据库**：NWP数据、站点历史数据、站点信息
- **Redis**：训练标记、格点开始时间缓存

### 8.2 模型存储

- **S3**：训练好的模型文件

### 8.3 输出数据

- **Kafka**：实时预测结果（消息队列）
- **数据库**：历史预测记录（用于回测）

## 11. 扩展性设计

- **时区扩展**：支持动态添加新时区，无需修改代码
- **模型扩展**：通过统一接口添加新模型类型
- **区域扩展**：自动发现新的经纬度格点
- **并行度调整**：根据CPU核心数和站点数量动态调整线程数
