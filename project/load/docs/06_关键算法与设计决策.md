# 关键算法与设计决策

## 1. 白名单机制的设计逻辑

### 1.1 为什么需要白名单？

XGBoost 是逐站训练和推理的，计算成本高（每站需要从 S3 下载模型、加载、特征工程、推理）。白名单机制筛选出"值得"用 XGBoost 的站点：

- **周期性强的站点**（ACF > 0.4）：XGBoost 的时间特征能有效捕获周期模式
- **负荷较高的站点**（avg_load ≥ 0.5）：预测误差的绝对影响更大
- **历史预测差的站点**（bad case）：需要更精细的模型

不在白名单的站点只使用 DLinear/TimeXer 的批量推理结果。

### 1.2 ACF 检验的含义

ACF(lag=2016) 表示当前时刻与7天前同一时刻的自相关系数。如果 > 0.4，说明该站点的负荷具有显著的**周周期性**，适合基于历史模式的预测。

## 2. select_best_7 集成策略

### 2.1 算法思路

不是简单的模型平均，而是**动态选择**：

1. 获取过去 N 天（Backtesting_days，默认7天）每个模型的预测结果
2. 与实际负荷对比，计算每个模型每天的**小时级 MAE**
3. 对每天选择 MAE 最低的模型
4. 最终输出 = 过去7天中表现最好的那个模型在目标日的预测

### 2.2 为什么用小时级 MAE？

- 5分钟级别的噪声太大，不适合评估模型整体趋势
- 小时级聚合后更能反映模型对负荷曲线形状的捕获能力
- 与业务关注的"小时电量"指标对齐

## 3. 节假日数据选择策略

### 3.1 Holiday-ON：选最低负荷7天

**假设**：节假日期间负荷模式类似于历史上负荷最低的日子（工厂停工、办公楼空置等）。

**实现细节**：
- 从60天历史中计算每天的平均负荷
- 排序后选择最低的7天
- 如果有效天数只有3-6天，通过复制策略扩展到7天
- 复制策略：从第2天开始循环复制（避免复制第1天，因为第1天可能是过渡日）

### 3.2 Holiday-OFF：遮蔽节假日数据

**假设**：节假日结束后，站点恢复正常模式，应该用正常模式的历史数据预测。

**实现细节**：
- 查询 `station_mode_operation_log` 获取所有节假日开启/关闭时间
- 构建时间掩码，标记所有节假日期间的数据点
- 从历史数据中删除被标记的点
- 要求剩余数据至少覆盖7天

## 5. 深度学习模型的输入输出设计

### 5.1 为什么输入2016点、输出648点？

- **输入 2016 = 7天 × 288点/天**：捕获完整的周周期
- **输出 648 = 2.25天 × 288点/天**：覆盖明天（288点）+ 后天（288点）+ 余量（72点）
- 实际使用时只取前 288 或 576 个点

### 5.2 为什么 load 要除以 100？

原始 load 值的量级在 0-100+ 范围（经过 `* 5/60 * 100` 转换后）。除以 100 将其归一化到 0-1 范围，与深度学习模型训练时的数据分布一致。

### 5.3 DLinear vs TimeXer 的标准化差异

- **DLinear**：在推理代码中手动做 instance normalization（减均值除标准差），然后反标准化
- **TimeXer**：模型内部自带 Non-stationary Normalization（`use_norm=True`），不需要外部处理

## 6. 线程安全设计

### 6.1 模型推理锁

深度学习模型不是线程安全的（共享 GPU 内存/模型参数），因此使用全局锁：

```python
_dlinear_inference_lock = threading.Lock()
_timexer_inference_lock = threading.Lock()
```

### 6.2 XGBoost 推理锁

XGBoost 推理涉及文件 I/O（S3 下载、pkl 加载），使用传入的 `lock` 参数保护：

```python
def batch_forecast_xgb_model(lock, batch_np, ...):
    lock.acquire()
    # ... 逐站推理 ...
    lock.release()
```

### 6.3 随机种子固定

每次推理前固定随机种子，确保结果可复现：
```python
random.seed(2021)
torch.manual_seed(2021)
np.random.seed(2021)
```

## 7. 夏令时/冬令时处理

### 7.1 问题描述

- **夏令时切换日**（如美国3月第二个周日）：2:00 直接跳到 3:00，缺少12个5分钟点
- **冬令时切换日**（如美国11月第一个周日）：1:00-2:00 重复出现，多出12个点

### 7.2 处理策略

- **夏令时**：从288个预测点中删除对应时段的点（输出 276 个点）
- **冬令时**：在对应时段插入重复的预测点（输出 300 个点）

## 9. 容错设计

### 9.1 多层 try-except

每个站点的处理都有独立的异常捕获，单站失败不影响其他站点：
```python
for station_id in station_list:
    try:
        predict(station_id)
    except Exception as e:
        logger.error(f"站点{station_id}预测失败")
        continue  # 继续处理下一个站点
```

### 9.2 备份预测

主预测（13:30）失败时，备份预测（22:00）提供第二次机会。

### 9.3 白名单安全检查

Bad Case 白名单有50%阈值保护，防止系统性异常导致所有站点被标记。
