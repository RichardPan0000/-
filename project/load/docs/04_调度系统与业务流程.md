# 调度系统与业务流程

## 1. 调度架构

所有调度任务继承 `threading.Thread`，由 `model_dispatch.py` 统一启动。每个任务内部使用 APScheduler 的 `BlockingScheduler` 实现定时触发。

### 1.1 调度时间线（以普通站点为例）

```
00:00  ┃ 白名单计算（周六）/ Bad Case 计算（周日）
01:15  ┃ XGBoost 模型训练
03:00  ┃ 最大负荷记录
       ┃
13:30  ┃ ★ 主预测（DLinear + TimeXer + XGBoost）
       ┃
18:00  ┃ 节假日开启模式预测
20:00  ┃ 节假日关闭模式预测
22:00  ┃ 备份预测
22:30  ┃ 补充预测
```

### 1.2 德国站点时间线

```
03:15  ┃ XGBoost 模型训练
16:30  ┃ ★ 主预测
23:00  ┃ 备份预测
```

## 2. 主预测流程（load_pred_schedule.py）

### 2.1 入口类：LoadModelPrediction

```python
class LoadModelPrediction(threading.Thread):
    def __init__(self, time_area, mode, exp_dlinear, exp_timexer, lock, 
                 global_white_list, region, trigger_hour, trigger_minute):
        ...
    
    def run(self):
        if self.mode == 2:
            self._run_once()      # 单次执行
        else:
            self._run_schedule()  # 定时调度
```

### 2.2 核心函数：pred_area_load_multi_model_parallel

```
pred_area_load_multi_model_parallel(time_area, exp_dlinear, exp_timexer, lock, global_white_list, region)
│
├── 1. 确定预测目标日期
│   └── pred_now_date = 当前本地时间（对齐到5分钟）
│       target_date = pred_now_date + 1天
│
├── 2. 检查夏令时/冬令时
│   ├── gap_time = get_loss_hour(time_area, target_date)     # 夏令时缺失
│   └── winter_extra_time = get_repeat_hour(time_area, target_date)  # 冬令时重复
│
├── 3. 启动多模型预测 Worker
│   └── predict_area_load_multi_model_parallel_worker(...)
│       │
│       ├── 3.1 批量获取站点数据
│       │   └── get_area_station_data_his_generator(area, batch_size=25, past_days_num=8)
│       │
│       ├── 3.2 数据预处理
│       │   └── prepare_data_for_multi_model(batch_df, station_mode_list, pred_now_date)
│       │       → numpy [N, 2016, 2]
│       │
│       ├── 3.3 DLinear 批量推理
│       │   └── process_batch_for_dlinear(batch_np, exp_dlinear)
│       │       → result [N, 648, 1]
│       │       → send_station_dlinear_result_to_kafka(result, ..., model='dlinear')
│       │
│       ├── 3.4 TimeXer 批量推理
│       │   └── process_batch_for_timexer(batch_np, exp_timexer)
│       │       → result [N, 648, 1]
│       │       → send_station_dlinear_result_to_kafka(result, ..., model='timexer')
│       │
│       └── 3.5 XGBoost 逐站推理（仅白名单站点）
│           └── batch_forecast_xgb_model(lock, batch_np, station_id_mode_list, ...)
│               → 从 S3 下载 pkl → 加载模型 → 特征工程 → 推理 → 写 Kafka
│
├── 4. 等待推理完成（sleep 10s）
│
├── 5. 最优模型选择
│   └── for station_id in ai_station:
│       select_best_model_hour_sum_v1_dst_wst(station_id, Backtesting_days, ...)
│       → 对比各模型历史精度 → 选择最优 → 写 Redis + Kafka
│
└── 6. 网格负荷预测（可选）
    └── predict_grid_load_parallel(time_area, pred_now_date, gap_time)
```

### 2.3 Kafka 输出格式

```
{pred_date};{station_id};{pred_type};{model};{version};{record_time};{statistics_time_list};{pred_value_list}

示例:
2025-05-13;12025010100001;2;dlinear;0.1;1715590000;[1715558400,1715558700,...];[0.12345,0.12456,...]
```

字段说明：
- `pred_date`: 预测目标日期
- `station_id`: 站点ID
- `pred_type`: 2（负荷预测）
- `model`: 模型名称（dlinear/timexer/GridXGB5fold-load-dayahead）
- `version`: 版本号
- `record_time`: 记录时间戳
- `statistics_time_list`: 预测时间点列表（Unix秒）
- `pred_value_list`: 预测值列表

## 3. 训练流程（load_train_schedule.py）

### 3.1 入口类：LoadModelTrain

- 触发时间：每天 01:15（普通）/ 03:15（德国）
- 每次执行前从 Redis 读取最新白名单

### 3.2 训练流程

```
train_area_load_parallel(time_area, white_list, region)
│
├── 1. 获取站点列表
│   └── 根据 region 获取普通站点或德国站点
│
├── 2. 批量获取历史数据
│   └── get_area_station_data_his_generator(area, batch_size=25, past_days_num=60)
│
├── 3. 逐站训练（线程池并发）
│   └── for station_id in batch:
│       ├── 获取站点历史数据（60天）
│       ├── ModelLoad().train_with_white_list(his_data, date, station_id, white_list)
│       │   ├── 特征工程 (feature_engineer_v2)
│       │   ├── 截取训练窗口（白名单50天/普通20天）
│       │   └── XGBoost 5折训练 + Top-3选择
│       └── 序列化模型参数 → 上传 S3
│
└── 4. 信号量控制并发数
    └── thread_train_limit_sem (配置中的线程限制)
```

### 3.3 模型上传

训练完成后，只上传模型的 `best_models` 参数列表（而非整个对象），减小文件体积：

## 4. 节假日预测

### 4.1 节假日开启模式（load_pred_holiday_on_schedule.py）

**场景**：站点进入节假日模式，负荷模式与平时不同（通常更低）。

**策略**：从过去60天历史中，选择**日均负荷最低的7天**作为输入，模拟节假日低负荷模式。

```
pred_area_load_multi_model_parallel_holiday(time_area, ...)
│
├── 1. 从 station_mode_operation_log 获取节假日站点列表
│
├── 2. 获取60天历史数据
│   └── get_area_station_data_his_generator_holiday(area, batch_size=25, past_days_num=60)
│
├── 3. 数据处理
│   └── prepare_data_for_multi_model_holiday()
│       ├── 计算每天的平均负荷
│       ├── 选择日均负荷最低的7天
│       ├── 如果有效天数 3-6 天，复制扩展到7天
│       └── 拼接为 2016 个点
│
├── 4. 仅使用 DLinear 推理（不用 XGBoost/TimeXer）
│
└── 5. 结果写 Kafka
```

### 4.2 节假日关闭模式（load_pred_holiday_off_schedule.py）

**场景**：站点从节假日模式恢复正常，需要排除节假日期间的异常数据。

**策略**：从历史数据中**遮蔽（mask）所有节假日时段**，只使用正常模式的数据进行预测。

```
LoadModelPredictionHolidayOff(time_area, ...)
│
├── 1. 获取节假日时间掩码
│   └── get_holiday_mode_mask(station_id)
│       → 查询 station_mode_operation_log 表
│       → 返回所有节假日开启/关闭的时间区间
│
├── 2. 应用掩码，删除节假日期间的数据
│   └── prepare_data_for_multi_model_holiday_off()
│       → 保留非节假日数据 → 要求至少7天有效数据
│
├── 3. DLinear + TimeXer + XGBoost 推理
│
└── 4. 结果写 Kafka
```

### 4.3 日内节假日触发（load_holiday_intra_on_call.py）

**场景**：站点在日内切换到节假日模式，需要立即产出新的预测。

**接口**：Flask HTTP API

```python
# POST /predict_holiday_on
# 请求体: {"station_ids": [...], "holiday_start_date": "2025-05-13"}
# 响应: {"status": "success", "task_id": "uuid"}
```

处理逻辑：
1. 接收请求后启动后台线程
2. 获取站点60天历史数据
3. 选择最低负荷的7天（3-6天时复制扩展）
4. DLinear 推理
5. 输出今天（日内剩余）、明天、后天的预测

## 5. 白名单系统

### 5.1 白名单计算（get_load_white_list.py）

**目的**：筛选出具有稳定周期性的站点，这些站点适合使用 XGBoost 进行精细化预测。

**触发**：每周六 00:00

**算法**：
```python
def cal_acf_for_batch_data(batch_df):
    for station_id, station_df in station_data_dict.items():
        load_series = station_df['load'].values
        
        # 条件1：数据长度 > 2016（至少7天）
        if len(load_series) <= 2016:
            continue
        
        # 条件2：平均负荷 >= 0.5 → 直接入白名单
        if load_series.mean() >= 0.5:
            white_list.append(station_id)
            continue
        
        # 条件3：ACF(lag=2016) > 0.4 → 周期性显著
        acf_values = acf(load_series, nlags=2016, fft=True)
        if acf_values[2016] > 0.4:
            white_list.append(station_id)
```

### 5.2 Bad Case 白名单（get_load_white_list_bad_case.py）

**目的**：找出预测效果差的站点，强制加入 XGBoost 训练。

**触发**：每周日 00:00

**算法**：
```sql
-- 过去30天中，mape_acc_1hour < 0.4 的天数 >= 4 的站点
SELECT station_id FROM station_cnt WHERE bad_dt_cnt >= 4
```

**安全检查**：如果 bad_case 数量 > 总站点数 × 50%，认为系统性异常，不上报。

### 5.3 白名单合并与存储

```
最终白名单 = ACF白名单 ∪ Bad Case白名单
存储位置: Redis key = 'load-pred:white_list_total'
```

## 6. 补充预测（load_pred_supplementary.py）

**场景**：对指定历史日期重新运行预测（如数据修复后需要补跑）。

**配置**：通过 Nacos 配置 `supplementary.date` 指定日期（支持逗号分隔多个日期）。

**触发**：每天 22:30 和 12:00

## 7. 过滤预测（load_pred_filter_schedule.py）

**场景**：某些站点在特定日期区间有异常数据（如设备故障），需要排除这些数据后重新预测。

**策略**：
1. 配置中指定站点和需要排除的日期区间
2. 从历史数据中删除指定区间
3. 从剩余数据中选择最近7个有效日期（每天 ≥280 个点）
4. 使用 DLinear 进行预测

## 8. 最大负荷记录（load_max_record_schedule.py）

**目的**：记录每个站点过去7天/14天/30天的最大负荷日数据，作为参考基线。

**触发**：每天 03:00

**输出模型名**：
- `max_load_last_one_week`：过去7天最大负荷日
- `max_load_last_two_week`：过去14天最大负荷日
- `max_load_last_one_mon`：过去30天最大负荷日

**输出**：同时写 Kafka 和 Redis，预测今天和明天的值（使用历史最大日的曲线形状）。
