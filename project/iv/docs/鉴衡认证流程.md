# 鉴衡 IV 认证流程说明

## 1. 文档目的

这份文档不是讲原有通用诊断流程，而是专门说明当前新增的“鉴衡 IV 认证”分析流程，目的是让后续使用时能直接知道：

- 这条流程和主诊断流程是什么关系
- 具体逻辑是什么
- 代码入口在哪里
- 命令怎么调用
- 输出文件怎么看
- 后续如果要接到更正式的产物里，应该从哪里改

当前实现已经落地到代码中，但它还是一个独立脚本，不是主 `orchestrator` 的正式 flow type。

## 2. 适用场景

这条流程对应的认证场景前提是：

- 输入是一整个站点在某个时刻的 IV 快照
- 每个 MPPT 下固定只有 `2` 条组串
- 两条串里通常一条是正常串，另一条是布置的故障串
- 对于 `Rs`、`Rsh`、`Voc`、阶梯/畸形分析，仍然希望沿用现有整站诊断逻辑
- 对于 `dIsc`、`dVoc`、`dImp`、`dVmp`、`dPmp`，不再和推导健康曲线比较，而是改成和同 MPPT 的另一条参考串比较

如果输入不满足“每个 MPPT 恰好 2 串”，当前实现会直接报错，不会自动降级。

## 3. 与现有主流程的关系

这条认证流程不是完全重写一套诊断，而是在现有站级诊断结果上再做一层“认证视角”的封装。

它复用的还是当前主基线：

- `healthy_curve` flow
- `RulePredictor4`
- `step_detect`

具体来说，认证流程先跑一遍现有整站诊断，然后从诊断结果里取这些已经算好的信息：

- `features`
- `voc_total_analysis`
- `slope_anomalies`
- `pattern_analysis`

在这个基础上，认证流程只额外改了一件核心事情：

- 最大功率相关和关键点偏移相关的对比对象，从“健康参考曲线”改成“同 MPPT 参考串”

所以它本质上是：

1. 先跑原有整站诊断
2. 再把输出转成鉴衡认证需要的 pairwise 表象结果

## 4. 代码位置

### 4.1 主实现

- `src/diagnosis/certification/jianheng_iv_certification.py`

## 5. 整体执行流程

整条流程可以按下面理解。

### 5.1 输入阶段

有两种入口：

#### 方式 A：CSV 模式

调用 `run_csv_certification(...)`。

输入是：

- `processed_df`
- `labels_df`
- `output_dir`

可选过滤条件：

- `task_id`
- `inverter_id`
- `timestamp`

CSV 模式里，代码会先做筛选，然后调用现有的 `build_diagnosis_batches(...)`，把数据整理成一个个站点快照批次，再逐批调用认证分析器。

#### 方式 B：Live 模式

调用 `run_live_certification(...)`。

输入是：

- `station_id`
- `sn_code`
- `dt`
- `output_dir`

这条路径现在分成两种情况：

1. 传了 `task_id`
   - 先调用 `diagnose_runtime_task_core(...)`
   - 这里会复用 runtime 的前半段逻辑：
     - 按 `task_id` 从数据库取数
     - `IVDataParser.parse_from_db_dataframe(...)`
     - `_build_runtime_environment_inputs(...)`
     - `orchestrator.diagnose_station(...)`
   - 然后把已经得到的 `station_data` 和 `diagnosis_results` 注入 `JianhengIVCertificationAnalyzer.analyze_station(...)`
2. 没传 `task_id`
   - 继续走原来的 `load_live_station_snapshot(...)`
   - 从数据库结果里解析出 `StationData`，再交给认证分析器

这里要特别注意：Jianheng 只复用了 runtime 的“取数 + 主诊断”前半段，没有复用 runtime 后面的 rulemap 改写、Kafka、`iv_result`、`iv_defect`、runtime 报告生成或结果落库逻辑。

## 6. 认证主逻辑

### 6.1 先跑原有整站诊断

`JianhengIVCertificationAnalyzer.analyze_station(...)` 里，第一步不是直接做 pairwise，而是先拿到原有整站诊断结果。

默认情况下，它会自己调用：

```python
self.orchestrator.diagnose_station(
    station_data=station_data,
    sn_code=sn_code,
    use_mppt_level=False,
    estimate_n_components=estimate_n_components,
)
```

现在这个入口还支持额外传入 `diagnosis_results`。

- 如果没有传 `diagnosis_results`
  - 行为和以前一样，由 `analyze_station(...)` 内部自己跑主诊断
- 如果传入了 `diagnosis_results`
  - 就跳过内部 `diagnose_station(...)`
  - 直接复用外部已经跑好的主诊断结果

`live + task_id` 现在走的就是第二种模式：先按 runtime 语义完成主诊断，再把结果注入进来做鉴衡后处理。

这里有两个重点：

- `use_mppt_level=False`
  - 说明仍然是整站上下文诊断，不是只看当前 2 串 MPPT
- `estimate_n_components`
  - 会继续透传给原流程

对于 `live + task_id` 这条新路径，还会额外透传 `runtime_environment_inputs`，也就是 runtime 从原始任务数据里提取的：

- `temp`
- `back_temp`
- `irradiance`

也就是说，`Rs/Rsh`、`Voc total`、阶梯/畸形这类依赖站级上下文的分析逻辑没有被切断。

同时也要明确边界：鉴衡后面的参考串选择、pairwise diff/hit、open-circuit 特判、CSV 输出、Word 报告、行级 rulemap enrich 都没有改成 runtime 风格，仍然完全沿用 Jianheng 自己的后处理逻辑。

### 6.2 从原诊断结果里取认证所需字段

每条串的诊断结果里，会取出这些结构：

- `features`
- `voc_total_analysis`
- `slope_anomalies`
- `pattern_analysis`
- `environment`

这一步通过 `_extract_prediction_payload(...)` 完成。

其中：

- `features` 里主要有 `Isc/Voc/Imp/Vmp/Pmp/FF`
- `voc_total_analysis` 保留原有总串 Voc 分析结论
- `slope_anomalies` 保留 `low_rsh/high_rs`
- `pattern_analysis` 保留 `step_abnormal/deform_abnormal`
- `pattern_analysis.steps` 里的 `pre_step/post_step` 可直接区分台阶主要落在 pre-MPP 还是 post-MPP

这里的 `post_step` 来自 `StepDetectPatternDetector` 的 post-MPP 后阶梯检测。当前默认 post 检测方法是 `optimal`：

- pre-step 检测逻辑保持不变
- post-step 默认使用 PAVA + 滑窗 slope + slope-of-slope 检测后阶梯电流下跳
- `post_step` 仍然由 `post_score >= 0.06` 得到
- 旧版 power-law + 二阶导 post 检测可通过 `post_detection_method: "legacy"` 回退

因此鉴衡输出里的 `post_step/post_score/post_step_locs` 字段名不变，但 post-step 的底层检测方法已经切换为 optimal post 检测。

### 6.3 校验每个 MPPT 必须恰好 2 串

认证逻辑在 MPPT 维度上有一个强前提：

```python
if len(mppt.strings) != 2:
    raise ValueError(...)
```

所以如果后续数据源有异常，例如少串、多串、串缺失，当前流程不会继续分析，而是直接失败。

### 6.4 参考串选择逻辑

参考串选择在：

- `select_reference_string(...)`

逻辑如下：

1. 对同一个 MPPT 的两条串，比较这三个指标：
   - `Isc`
   - `Voc`
   - `Pmp`
2. 哪条串在某个指标上更大，就拿到这一票
3. 三项多数票获胜者就是参考串

例如：

- A 的 `Isc` 更大
- B 的 `Voc` 更大
- A 的 `Pmp` 更大

那么 A 得 2 票，B 得 1 票，A 为参考串。

#### 平票时怎么处理

如果出现平票，代码会按下面顺序打破平票：

1. `Pmp` 更高者优先
2. `Voc` 更高者优先
3. `string_id` 字典序更小者优先

这部分是当前实现里写死的逻辑。

### 6.5 Pairwise diff 计算逻辑

pairwise diff 在：

- `build_pairwise_diffs(...)`

输出字段是：

- `dIsc_pair`
- `dVoc_pair`
- `dImp_pair`
- `dVmp_pair`
- `dPmp_pair`
- `dFF_pair`

计算方式是：

```python
(measured - reference) / abs(reference)
```

也就是：

- 比参考串低时，结果为负
- 比参考串高时，结果为正

#### 参考串这一行怎么写

如果当前串本身就是参考串，那么这 6 个 pairwise diff 都固定写成 `0.0`。

这点是故意这么做的，方便后续看结果时不需要再单独判断参考串。

### 6.6 Pairwise hit 布尔值

pairwise diff 算完后，还会进一步根据阈值输出一组布尔字段：

- `hit_disc_drop`
- `hit_dvoc_drop`
- `hit_dimp_drop`
- `hit_dvmp_drop`
- `hit_dpmp_drop`
- `hit_dff_drop`
- `hit_low_mpp_dpmp_drop`

这些不是重新定义一套阈值，而是直接读取当前 flow 中 `rule_predictor.rule4_thresholds`。

也就是说，这些 pairwise hit 仍然跟当前 `RulePredictor4` 的阈值配置保持一致，只是“比较对象”变了。

## 7. 哪些逻辑保持不变

这部分很关键，因为认证流程不是把所有逻辑都改成 pairwise。

### 7.1 `Rs/Rsh` 逻辑不变

当前认证流程没有把 `Rs/Rsh` 改成“只看 2 串局部差异”，而是继续沿用原有整站诊断算出来的：

- `slope_anomalies.low_rsh`
- `slope_anomalies.high_rs`

最终输出到结果表里的字段是：

- `is_low_rsh_slope_anomaly`
- `is_high_rs_slope_anomaly`

### 7.2 `Voc` 总串分析不变

`Voc` 相关的总串分析继续直接复用原输出：

- `voc_total_available`
- `voc_total_outlier`
- `dVoc_total`
- `comparison_total`
- `voc_total_analysis_json`

所以这部分仍然是原来那套总串 Voc 判定，不是简单地改成 pairwise `dVoc_pair`。

### 7.3 阶梯和畸形逻辑不变

阶梯检测和畸形检测继续沿用原有 `pattern_analysis`：

- `step_abnormal`
- `deform_abnormal`
- `num_steps`
- `max_step_depth`
- `pattern_analysis_json`

也就是说：

- 阶梯检测还是原来的方法
- 没有因为认证场景就切换成 pairwise 检测

## 8. Processed IV 导出逻辑

这条流程除了结果表，还会给每条串额外导出 processed IV，方便后续人工查看和留档。

### 8.1 导出内容

每条串会导出一个 CSV，列名固定为：

- `voltage_unit_interp`
- `current_unit_smooth`

### 8.2 数据来源

导出的不是原始 IV，而是复用了主流程里的处理思路：

1. 取原始串级 IV
2. 根据 `n_components` 转成单组件电压尺度
3. 调 `curve_processor.interpolate(...)` 插值
4. 调 `curve_processor.smooth(...)` 平滑

代码在：

- `_build_processed_curve_payload(...)`
- `_write_processed_curve(...)`

### 8.3 `n_components` 的处理

这一步也尽量和主流程保持一致：

- 优先用串对象上的 `n_components`
- 如果 flow 支持 `_get_portrait_for_string(...)` 和 `_resolve_n_components_for_diagnosis(...)`
  - 则继续按主流程的画像和估算逻辑解析 `n_components`
- `estimate_n_components` 参数也会透传进来

这意味着：

- 导出的 processed IV 和诊断时使用的组件数口径尽量保持一致

## 12. 使用时需要注意的点

### 12.1 这不是最终根因标签输出

当前流程只输出“认证表象分析结果”，还没有生成最终 `DiagnosisResult`，也没有做根因标签映射。

所以它当前更适合：

- 认证结果留档
- 人工复核
- 后续规则映射
- 离线分析

### 12.2 `live + task_id` 只复用 runtime 前半段

## 18. Rulemap Knowledge Graph Integration

当前 `run_jianheng` 流程已经在最终输出阶段接入了 `rulemap` 知识图谱推理层。

这层逻辑的目标不是替换前面的健康曲线诊断、pairwise 指标计算或图表生成，而是在这些结果已经生成之后，把现有表象字段送入 `rulemap`，输出一份面向认证解释的知识图谱推理结果。

如果需要理解 rulemap 如何从异常表象推断最终根因，优先看：

- `docs/rulemap_diagnosis_logic.md`

### 18.1 `rulemap` 是什么

`rulemap` 不是简单的 label map，也不是把某个字段硬映射成故障标签。

它是一套独立的知识图谱推理引擎，主要由下面两个文件驱动：

- `src/rulemap/knowledge_graph.yaml`
- `src/rulemap/diagnose.py`

`knowledge_graph.yaml` 里定义了：

- 表象节点，例如 `dIsc`、`dVoc`、`dPmp`、`Rs`、`Rsh`、`prestep`、`poststep`
- 根因节点，例如遮挡、PID、串联电阻增大、并联电阻下降、开路相关故障
- 表象到根因的加权边
- 根因反向验证所需的期望表象
- 表象模式，例如电流型、电压型、台阶型、串联电阻型、并联电阻型、开路型
- 矛盾规则，例如开路表象排除普通电流/功率下降、台阶表象排除 `dPmp`、`Rsh` 优先于 `Rs`

`diagnose.py` 的推理流程大致分四层：

1. 表象检测：负向偏差超过阈值才算异常，正向偏差视为正常。
2. 矛盾规则修正：根据知识图谱里的排除、优先级和警告规则修正表象集合。
3. 正向推理：异常表象按边权重投票到候选根因。
4. 反向验证：用根因的期望表象计算覆盖率和置信度，并输出推理链路。

最终输出包含：

- `phenomena`
- `faults`
- `pattern_groups`
- `warnings`
- `reasoning`

### 18.2 Jianheng 到 rulemap 的输入映射

Jianheng 流程中的 pairwise 指标和已有诊断信号会在 `src/analysis/jianheng_rulemap_bridge.py` 中转换成 rulemap 输入。

当前映射关系是：

| Jianheng 输出字段 | rulemap 输入字段 |
| --- | --- |
| `dIsc_pair` | `dIsc` |
| `dVoc_pair` | `dVoc` |
| `dImp_pair` | `dImp` |
| `dVmp_pair` | `dVmp` |
| `dPmp_pair` | `dPmp` |
| `dFF_pair` | `dFF` |
| `is_high_rs_slope_anomaly` | `Rs` |
| `is_low_rsh_slope_anomaly` | `Rsh` |
| `pre_step` | `prestep` |
| `post_step` | `poststep` |

这意味着 rulemap 的数值类表象使用的是同 MPPT 参考串的 pairwise 偏差，而不是主诊断流里和健康参考曲线比较得到的 `dIsc/dVoc/...`。

`voc_total_analysis`、`step_abnormal`、`deform_abnormal`、`scan_status` 当前没有直接作为 rulemap 原生输入节点进入知识图谱。

### 18.3 接入位置和输出字段

接入点在：

- `src/diagnosis/certification/jianheng_iv_certification.py`
- `_write_combined_outputs(...)`

也就是说，rulemap 是在每个 batch 的 `rows` 已经生成之后、写出 `certification_surface_results.csv` 之前执行的。这样 CSV 模式和 live 模式都会走同一套 rulemap 输出逻辑。

`certification_surface_results.csv` 会新增这些字段：

- `rulemap_status`
- `rulemap_primary_pattern`
- `rulemap_primary_cause`
- `rulemap_primary_confidence`
- `rulemap_final_phenomena`
- `rulemap_final_causes`
- `rulemap_final_causes_json`
- `rulemap_phenomena_json`
- `rulemap_faults_json`
- `rulemap_pattern_groups_json`
- `rulemap_warnings_json`
- `rulemap_reasoning`
- `rulemap_error`

其中 `rulemap_final_phenomena` 对齐原始 rulemap 输出里的“异常表象”，例如：

```text
dVmp=-0.2193, dFF=-0.2462, Rs=+1.0000, poststep=+1.0000
```

`rulemap_final_causes` 是最终三个根因列，取法参考 `src/rulemap/run_diagnosis.py`：按 `faults` 顺序扁平化所有 `fault.causes`，再取前三个，并保留 `score/conf/source` 信息。示例：

```text
串联电阻增大(score=14.0181,conf=1.00,src=dFF|dVmp); 连接器接触不良(score=1.0000,conf=1.00,src=Rs); 线缆老化(score=0.8000,conf=1.00,src=Rs)
```

`rulemap_final_causes_json` 是这三个最终根因的结构化版本，适合后续程序分析。

`rulemap_primary_pattern`、`rulemap_primary_cause`、`rulemap_primary_confidence` 只是为了兼容和粗筛选保留的摘要字段，不应再被当成完整最终根因解释。完整多故障过程仍以 `rulemap_faults_json` 为准。

`summary.json` 会新增这些统计字段：

- `rulemap_enabled`
- `rulemap_graph_path`
- `rulemap_status_counts`
- `rulemap_primary_pattern_counts`
- `rulemap_primary_cause_counts`

### 18.4 Open Circuit 的特殊处理

当前 `scan_status=2/3` 或原诊断结果为 `Open_Circuit` 的行，不走普通 rulemap 推理，而是在 bridge 里做确定性 override。

原因是：

- 当前 rulemap 主要吃 pairwise 偏差和 `Rs/Rsh/prestep/poststep`
- `scan_status` 还不是知识图谱里的原生节点
- 开路在 Jianheng 流程里已经有明确的业务规则

因此这类行会输出：

- `rulemap_status = overridden_open_circuit`
- `rulemap_primary_pattern = open_circuit_override`
- `rulemap_primary_cause = Open_Circuit`
- `rulemap_final_phenomena = Isc_near_zero=-1.0000`
- `rulemap_final_causes = Open_Circuit(score=1.0000,conf=1.00,src=Isc_near_zero)`

这保证了开路样本不会因为 pairwise 指标缺失或异常而得到不稳定的图谱结论。

### 18.5 当前差距和后续建议

当前接入已经满足“在 `run_jianheng` 末尾增加 rulemap 推理结果”的目标，但还不是完整的产品级 rulemap 主导输出。

当前差距：

- Word 报告还没有展示 rulemap 结果。
- 每个 MPPT 目录下的 `pair_metadata.json` 还没有写入 rulemap 结果。
- 旧字段 `diagnosis` 仍然保留主诊断流结果，没有被 rulemap 替换。
- `Open_Circuit` 是 bridge 层 override，不是知识图谱原生推理。
- `voc_total_outlier`、`dVoc_total`、`step_abnormal`、`deform_abnormal`、`scan_status` 还没有作为知识图谱节点建模。
- rulemap 使用 `knowledge_graph.yaml` 中自己的阈值，例如 `dIsc/dVoc/dPmp=0.05`、`dFF=0.02`，不一定和 `RulePredictor4` 当前阈值完全一致。

如果后续要让 rulemap 成为认证结果的主视角，建议优先做两件事：

1. 把 `scan_status`、`voc_total_outlier`、`dVoc_total`、`step_abnormal/deform_abnormal` 纳入知识图谱输入模型。
2. 把 rulemap 的摘要结果写进 Word 报告，避免只看报告时漏掉最终推理结论。
